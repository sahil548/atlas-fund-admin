### Observation 20 — Asset-level calculations are the foundation of the entire return tree (cross-cutting)

**Page / Action:** /assets/[id] — Income, Expenses, Performance tabs (and the calculation chain downstream)
**What I saw:** User's strategic note: "These calculations for assets are also really important to get right. Because these are the leaves of the tree. The entity calculations, in terms of returns, investor returns, all the way up the chain are based on the asset returns that are here."

**Why this matters for triage:** This re-weights several prior Asset observations. Anything that affects the CORRECTNESS of asset income / expenses / valuations cascades up to:
- Entity (vehicle) NAV, TVPI, DPI, RVPI
- Fund-level IRR and MOIC
- LP-level distributions, capital account statements, K-1s
- Waterfall calculations (pref accrual depends on accurate return streams)

Observations that now graduate in priority because they affect calculation quality:
- **Obs 10** (can't edit entry date + other fields) — if a correction is needed and the user can't make it, the entire downstream tree is stuck with wrong numbers.
- **Obs 12** (no lease/contract edit path) — contracts drive the expected cash flow model.
- **Obs 13** (need daily granularity) — end-of-quarter snapshots lose precision on XIRR.
- **Obs 15 + 16** (AI-drafting from contracts) — reframes from "nice to have" to "reduces manual data entry errors that propagate upward."

Less urgent reframing needed for:
- **Obs 17** (document categories) — display only, no calc impact.
- **Obs 18** (task linking) — workflow ergonomics, no calc impact.

This observation is META — it doesn't need its own fix; it informs the triage of Obs 10/12/13/15/16. Claude should weight those more heavily during Wave 3.

**Severity:** n/a (strategic weighting, not a bug)

### Observation 35 — "Add Side Letter" form CRASHES with runtime TypeError — BLOCKER

**Page / Action:** /directory → "Add Side Letter" button → form crashes on open
**What I saw:** Runtime TypeError: `(entities || []).map is not a function or its return value is not iterable` at [`create-side-letter-form.tsx:150`](atlas/src/components/features/side-letters/create-side-letter-form.tsx). Root cause diagnosed in code: the form fetches `/api/entities?firmId=X` and uses `r.data ?? r` to unwrap the response. During v2.1 the `/api/entities` endpoint response shape appears to have changed (likely paginated or object-wrapped), so `entities` is now an object rather than an array; `(entities || []).map` fails because truthy-but-non-array objects skip the `|| []` fallback. This form was not updated to match the new shape. Same shape issue almost certainly applies to the `investors` fetch on line 53 — that will crash the moment entities isn't the first thing evaluated.

**Fix scope:** either (a) normalize `/api/entities` and `/api/investors` endpoints to always return an array at the top level, or (b) update this form (and every similar consumer) to explicitly unwrap `.items` / `.data` safely. Option (a) is the conservative path; option (b) is the one that unblocks v3.0 Phase 25 (pagination) without another forced response-shape migration.

**Severity:**
- [x] Blocker — user cannot add side letters at all; entire flow is 404-equivalent
- [ ] Annoying
- [ ] Nitpick

**Claude Triage:** *(Wave 3 — very likely Urgent → Phase 22 FIN-08 or new FIN item; side-letter infrastructure is explicitly in scope per PROJECT.md "Side letters: Extensive, per-LP, per-fund. Common in family office.")*

3. If you had to fix ONE thing before using this for real (for an actual LP or real deal), what would it be?

**Asset-level correctness and editability.** Direct quote: "the one thing we absolutely need to get right is the assets because everything else rolls up from there." Entity NAV, investor returns, fund-level IRR, waterfall accrual, LP K-1s — all roll up from asset-level income, expenses, and valuations. If the asset layer has incorrect or uneditable data, every downstream number inherits that error.

This single answer re-prioritizes the Phase 22 scope: the "calc-critical" asset observations (Obs 10 edit gaps, Obs 12 lease/contract edit path, Obs 13 daily granularity, Obs 25 cost basis seed gap, Obs 26 unit-class terms) move from "annoying" to "highest-priority fit-and-finish work" for v3.0.
