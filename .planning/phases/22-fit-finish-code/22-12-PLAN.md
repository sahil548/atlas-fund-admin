# Plan 22-12: Multi-Entity Allocation on Create + Number-Field Guardrails

<objective>
Close punch-list gap #5 (multi-entity allocation during Add Asset) and the structured-field half of #7 (HTML min/max on number inputs).

Today: Add Asset only takes one entity and one allocation %. A real asset often sits across multiple fund vehicles — Fund II + Fund III Co-Invest SPV, for example, at a 60/40 split. Users can do this via the per-entity allocation edit flow after the asset is created, but having to create-then-edit is onboarding friction.

Scope intentionally narrow — single plan, atomic commit. Defers:
- Type-conditional detail-field validation (those fields are stored as free-text strings today — real guardrails need a type change, which deserves its own plan).
- Multi-entity allocation editing on the Edit Asset modal (the existing asset-overview-tab allocation edit surface covers the edit path; Add was the acute gap).
</objective>

<must_haves>
<truths>
- `CreateAssetSchema` accepts an `allocations: Array<{ entityId, allocationPercent }>` field alongside the legacy single-entity shape for back-compat
- Schema refinement rejects: allocations that don't sum to 100 ± 0.01, duplicate entityIds
- `POST /api/assets` creates one `AssetEntityAllocation` row per allocation, with per-row `costBasis = totalCostBasis × allocationPercent/100`
- Legacy callers supplying only `entityId` + `allocationPercent` still work (single allocation)
- Add Asset form renders an "Entity Allocations *" fieldset with repeatable rows, + Add entity button, running total indicator, and a red "must sum to 100" warning when the total is off
- Client-side submit validation matches server-side (sum = 100, no duplicates, at least one row)
- HTML `min` / `max` attributes added to Ownership %, Share Count, Projected IRR, Projected Multiple on both Add and Edit forms — prevents scroll-wheel / spinner from crossing sane bounds
- `npm run build` passes with zero TypeScript errors
</truths>
</must_haves>
