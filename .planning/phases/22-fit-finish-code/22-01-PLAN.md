  <action>
Read both form files. The crash originates in `create-side-letter-form.tsx` where the SWR fetcher does `r.data ?? r` — if the API returns a paginated envelope whose `.data` is somehow missing or an object, `entities` becomes truthy-but-non-array and `.map` throws.

Fix by replacing every `r.data ?? r` unwrap with a strict `Array.isArray`-guarded helper. Concretely:

1. Near the top of `create-side-letter-form.tsx` (below imports), add a small inline helper:
   ```typescript
   const unwrapArray = <T,>(r: any): T[] => {
     if (Array.isArray(r)) return r;
     if (Array.isArray(r?.data)) return r.data;
     return [];
   };
   ```
2. Replace the two SWR fetchers so `investors` and `entities` are always arrays:
   ```typescript
   const { data: investors = [] } = useSWR<Investor[]>(
     open ? `/api/investors?firmId=${firmId}` : null,
     (url) => fetcher(url).then(unwrapArray<Investor>),
   );
   const { data: entities = [] } = useSWR<Entity[]>(
     open ? `/api/entities?firmId=${firmId}` : null,
     (url) => fetcher(url).then(unwrapArray<Entity>),
   );
   ```
3. Also make every downstream `.map` site defensive — any remaining `(entities || []).map` becomes plain `entities.map` (safe now because the SWR default is `[]`).
4. Repeat the same two-fetcher substitution in `edit-side-letter-form.tsx` — it has the same pattern and will crash for the same reason on any edit flow.

Do NOT change the API routes. Pagination is a Phase 25 concern; Phase 22's contract is "whatever shape the API returns today, the form tolerates it."

Follow coding-patterns.md: no `toast.error(data.error)` with unchecked objects (existing pattern is already fine); client components keep their `"use client"` directive.
