# Plan 22-10: Asset Edit / Add Parity (Post-Deploy Gap Closure)

<objective>
Close two gaps discovered AFTER Phase 22 shipped to production (2026-04-17, commit d754f14):

1. **Existing assets don't show the type-conditional fieldset in Edit Asset.** Production assets created before the `AssetRealEstateDetails` / `AssetCreditDetails` / `AssetEquityDetails` / `AssetFundLPDetails` child records were populated return null from `detectAssetKind`, so the fieldset renders as nothing. Users see the new Entry Date / Projected IRR fields but NOT the per-type scalars — which is the more important half of Obs 10.

2. **Add Asset form is missing every Phase 22 addition.** Entry Date, Projected IRR, Projected Multiple, and the type-conditional fieldset are only in the Edit modal. Creating a new asset from scratch goes through a narrower form, which means new users can't populate historical-date + per-type fields during onboarding — exactly the friction Obs 10 called out.

Bundled as 22-10 under Phase 22's "fit & finish" umbrella per user decision on 2026-04-17 ("this all needs to be part of phase 22"). Not a new phase.
</objective>

<must_haves>
<truths>
- Opening Edit Asset on an asset WITHOUT a type-specific child detail record shows the correct type-conditional fieldset based on asset class / capital instrument / participation structure
- Saving field values from that fieldset creates the AssetRealEstateDetails / AssetCreditDetails / AssetEquityDetails / AssetFundLPDetails row on first write (upsert), instead of returning a 400 "Type mismatch" error
- Add Asset form exposes Entry Date (defaults to today), Projected IRR (%), Projected Multiple (x), AND a type-conditional fieldset that tracks the selected Asset Class / Instrument / Participation
- POST `/api/assets` accepts the new `entryDate`, `projectedIRR`, `projectedMultiple`, `typeDetails` fields and persists them, creating the nested detail record when `typeDetails` is provided
- `npm run build` passes with zero TypeScript errors
</truths>
</must_haves>
