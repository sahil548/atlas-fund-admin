<objective>
Ship the strategic core of Phase 22: asset correctness. Per Obs 20 thesis ("Get assets right — everything rolls up from there"), this is the single most important plan in the phase. Close Obs 10 (entry date + other fields not editable), Obs 12 (no clear edit path for lease/contract fields), Obs 24 (Asset Allocations percentage ambiguity), and Obs 25 (entity overview cost bases zero).

Per CONTEXT.md locked decisions:
- **Every asset field editable**, including entry date (critical for onboarding historical assets).
- **Modal pattern** for the main Edit Asset form — not inline edit.
- **One modal, type-conditional SECTIONS in the main body.** Common fields (name, entry date, cost basis, class, entity, status) at top. Per-type section below renders only the scalar fields relevant to this asset's type — reading and writing back to the one-to-one type-specific details record (`AssetRealEstateDetails` for RE property specifics; `AssetCreditDetails` for credit facility scalars; `AssetEquityDetails` for ownership + EBITDA; `AssetFundLPDetails` for commitment + vintage).
- **Dedicated sub-modals for child rows.** Leases, contracts (CreditAgreements), valuations are ONE-TO-MANY with asset. They are edited via their own per-row Edit forms triggered from the Overview tab. This is distinct from the type-conditional section in the main Edit Asset modal, which edits the single one-to-one details record.

**Why the distinction matters.** A Private Credit asset has:
- One `AssetCreditDetails` row (one-to-one) with scalars like facility amount, interest rate / spread, maturity date, seniority. Edited in the MAIN Edit Asset modal's type-conditional section.
- Many `CreditAgreement` rows (one-to-many) with document-level terms. Edited via the `EditCreditAgreementForm` sub-modal from the Overview tab.
- Many `Covenant` and `CreditPayment` rows (one-to-many). Viewed on Overview; edit UI for these is out of scope here unless trivially close.

A Real Estate asset has:
- One `AssetRealEstateDetails` row (one-to-one) with scalars like property type, cap rate, square footage. Edited in the MAIN Edit Asset modal's type-conditional section.
- Many `Lease` rows (one-to-many). Edited via the `EditLeaseForm` sub-modal.
- Many `Valuation` and `AssetValuation` rows (one-to-many). Edited via the `EditValuationForm` sub-modal.

Purpose: This is onboarding-capability-level, not polish. A user bringing historical assets in cannot backdate them without entry-date edit access; a user correcting lease terms cannot currently do so at all; a user fixing a Private Credit asset's spread cannot currently do so at all (the scalar details are read-only today); an entity overview that shows zero cost bases looks broken even when data is present. The asset-correctness cluster is also the suspected root cause of LP-Obs 2's data-quality cascade (confirmed or refuted by Plan 22-03's diagnosis).

Output: A complete Edit Asset experience — main modal with common scalars + type-conditional scalar section for per-type details, sub-modals for one-to-many child rows, tooltip + correct display on entity overview, all wired to PUT endpoints that persist.
