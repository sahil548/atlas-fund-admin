```typescript
const totalDeals = deals.length;
const pastScreening = deals.filter(d => ["DUE_DILIGENCE","IC_REVIEW","CLOSING","CLOSED"].includes(d.stage)).length;
const pastDD = deals.filter(d => ["IC_REVIEW","CLOSING","CLOSED"].includes(d.stage)).length;
const pastIC = deals.filter(d => ["CLOSING","CLOSED"].includes(d.stage)).length;

const screeningToDD = totalDeals > 0 ? Math.round((pastScreening / totalDeals) * 100) : 0;

const icToClose = pastDD > 0 ? Math.round((pastIC / pastDD) * 100) : 0;
```
Note: These are cumulative counts, not stage-specific. DEAD deals are included in totalDeals but excluded from pastScreening, which means screeningToDD can never exceed 100%. But ddToIC and icToClose also cannot exceed 100% because pastIC <= pastDD <= pastScreening by definition. The 300% bug may have been from a DIFFERENT calculation that was already fixed, OR from a different display location. Investigate carefully.

```typescript
const totalTasks = workstreams.reduce((sum, ws) => sum + (ws.totalTasks ?? 0), 0);
const completedTasks = workstreams.reduce((sum, ws) => sum + (ws.completedTasks ?? 0), 0);
const overallPct = totalTasks > 0 ? Math.round((completedTasks / totalTasks) * 100) : 0;
```

The DD tab calculates from workstream data. If workstreams have no tasks (totalTasks=0), overallPct=0. This IS the bug: for deals past DD stage, workstreams may exist but tasks were never created (AI analysis creates them). The 0% display is technically correct for zero tasks, but misleading. The fix may be to show workstream status instead of task count, or to show "N/A" when no tasks exist.
