1. First, grep all console.log/warn/error usages in src/lib/:
   - `grep -rn "console\.\(log\|warn\|error\)" src/lib/ --include="*.ts" | grep -v "__tests__" | grep -v "node_modules"`
   - This identifies every call that needs migration

2. For EACH file with console.* calls, add `import { logger } from "@/lib/logger";` at the top and replace:
   - `console.error("message", data)` -> `logger.error("message", { ...relevant_data })`
   - `console.warn("message", data)` -> `logger.warn("message", { ...relevant_data })`
   - `console.log("message", data)` -> `logger.info("message", { ...relevant_data })` OR `logger.debug("message", { ...relevant_data })`
   - Use `logger.error` for caught exceptions and failure conditions
   - Use `logger.warn` for degraded behavior (e.g., missing config, fallbacks)
   - Use `logger.info` for significant events (e.g., "AI extraction started", "deal stage changed")
   - Use `logger.debug` for detailed diagnostic info (e.g., request payloads, intermediate calculations)

3. Migration rules:
   - Do NOT change the surrounding logic — only replace the console call
   - If the console.error was inside a catch block, keep it as logger.error
   - If the console.log was a debug/trace message, use logger.debug
   - If the console.warn was about missing config/fallback, keep as logger.warn
   - NEVER log sensitive data (API keys, passwords, PII) — if existing console.log contains sensitive data, redact it
   - For error objects, log `{ error: e.message }` or `{ error: String(e) }` — not the raw error object

4. Files to migrate (from RESEARCH.md — 111 files total, ~210 calls; this task covers src/lib/ core files):

   - `src/lib/dd-analysis-service.ts` — DD analysis
   - `src/lib/rate-limit.ts` — rate limiting
   - Any other files in src/lib/ that have console.* calls (check with grep)
