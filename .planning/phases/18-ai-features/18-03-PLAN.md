    **1. Add `planAction()` function to `src/lib/ai-service.ts`:**

    ```typescript
    export async function planAction(
      action: string,
      firmId: string,
      userId: string,
      pageContext?: { pageType: string; entityId?: string; entityName?: string }
    ): Promise<ActionPlan>
    ```

    This function:
    - Calls `createUserAIClient(userId, firmId)` — if null, return error plan
    - Gathers minimal context via `gatherContext(firmId)` (reuse existing)
    - Builds a system prompt specifically for action parsing (see RESEARCH.md Pattern 3):
      ```
      You are Atlas AI executing an action. Parse the user's request and return a JSON action plan.

      Available action types: CREATE_TASK, CREATE_DEAL, UPDATE_DEAL, LOG_NOTE, ASSIGN_TASK, CREATE_CAPITAL_CALL, CREATE_DISTRIBUTION, CREATE_MEETING, TRIGGER_DD_ANALYSIS, TRIGGER_IC_MEMO, EXTRACT_CIM_TERMS, DRAFT_LP_UPDATE, SUGGEST_TASKS

      Current page context: {pageContext if available}

      Available deals: {top 10 deals by name for ID resolution}
      Available entities: {top 10 entities by name}
      Available users: {team members for assignment}

      For each action return:
      {
        "actionType": "...",
        "description": "Human-readable description of what will happen",
        "payload": { /* relevant fields */ }
      }

      If the request is ambiguous (e.g., multiple people named Sarah), return:
      { "actionType": "AMBIGUOUS", "description": "clarification question", "payload": {} }
      ```
    - Uses `jsonrepair` for parsing LLM response (import from "jsonrepair", already in project)
    - Returns the parsed ActionPlan

    The context query for name resolution should be lightweight — fetch top deals, entities, and team members by name only (not full records). Use `prisma.deal.findMany({ where: { firmId }, select: { id: true, name: true }, take: 20 })` etc.

    **2. Create `src/app/api/ai/execute/route.ts`:**

    POST endpoint with two modes:
    - **Unconfirmed** (`confirmed: false` or absent): Calls `planAction()`, returns the ActionPlan for user review
    - **Confirmed** (`confirmed: true`): Executes the action based on actionType + payload

    Request schema (use Zod + parseBody):
    ```typescript
    const ExecuteSchema = z.object({
      action: z.string().min(1),
      firmId: z.string().min(1),
      pageContext: z.object({
        pageType: z.string(),
        entityId: z.string().optional(),
        entityName: z.string().optional(),
      }).optional(),
      confirmed: z.boolean().optional().default(false),
      confirmedPayload: z.record(z.unknown()).optional(),
      actionType: z.string().optional(),  // Only when confirmed=true
    });
    ```

    **Execution handlers by actionType (when confirmed=true):**

    - `CREATE_TASK`: POST to internal `/api/tasks` with payload fields (title, priority, contextType, contextId, assigneeId). Use `fetch()` with absolute URL.
    - `CREATE_DEAL`: POST to internal `/api/deals` with payload fields (name, description, etc.)
    - `UPDATE_DEAL`: PUT to internal `/api/deals/[id]` with payload fields
    - `LOG_NOTE`: POST to internal `/api/deals/[id]/notes` or similar activity endpoint
    - `TRIGGER_DD_ANALYSIS`: POST to internal `/api/deals/[id]/dd-analyze` with `{ type: "DD_FINANCIAL" }`. Return immediately with a message "DD analysis started — this may take up to 60 seconds. Check the DD tab on the deal page."
    - `TRIGGER_IC_MEMO`: POST to internal `/api/deals/[id]/dd-analyze` with `{ type: "IC_MEMO" }`. Same long-running message.
    - `EXTRACT_CIM_TERMS`: POST to internal `/api/documents/[id]/extract` — trigger re-extraction on a specific document
    - `ASSIGN_TASK`: PUT to internal `/api/tasks/[id]` with assigneeId
    - `AMBIGUOUS`: Return the clarification question to the user (no execution)

    For internal API calls, construct the URL using `process.env.NEXT_PUBLIC_APP_URL || "http://localhost:3000"` as the base. Pass through the auth headers from the original request.

    Apply rate limiting consistent with `/api/ai/search` (use same `rateLimit()` pattern).

    Use `getAuthUser()` for userId. Use `createUserAIClient(userId, firmId)` for AI calls (Phase 12 pattern).

    **Key anti-patterns to avoid:**
    - Do NOT auto-execute without confirmation (always return plan first when confirmed=false)
    - Do NOT expose raw DD endpoint to command bar — go through /api/ai/execute
    - Do NOT block on DD/IC memo generation — return "started" message immediately

    **1. Create `src/components/features/command-bar/action-confirmation.tsx`:**

    Component that renders within the command bar dropdown (or side panel) when an AI action plan is returned.

    ```typescript
    "use client";

    interface ActionConfirmationProps {
      plan: ActionPlan;
      onConfirm: (editedPayload: Record<string, unknown>) => void;
      onCancel: () => void;
      isExecuting: boolean;
    }
    ```

    UI within the command bar dropdown area:
    - Shows `plan.description` as a summary of what will happen
    - Renders editable fields based on `plan.payload` — each key/value pair as a labeled input field
      - String values: text input
      - Boolean values: checkbox
      - "priority" key: select dropdown with HIGH/MEDIUM/LOW
      - Read-only for IDs (entityId, dealId) — show but don't allow editing
    - Two buttons: "Confirm" (primary blue) and "Cancel" (gray)
    - Loading state when `isExecuting` (disable buttons, show spinner)
    - For `AMBIGUOUS` actionType: show the description as a question to the user, with a text input for clarification (no confirm button — user types answer and it re-submits as a new action)
    - For `TRIGGER_DD_ANALYSIS` and `TRIGGER_IC_MEMO`: show a note "This may take up to 60 seconds" in the description area

    Style with Tailwind, matching existing command bar styling (bg-white, rounded, shadow, dark mode variants).

    **2. Wire nl_action intent handling in `command-bar.tsx`:**

    When `classifyIntent()` returns `"nl_action"`:
    1. Call `/api/ai/execute` with `{ action: inputText, firmId, pageContext, confirmed: false }`
    2. Receive ActionPlan response
    3. Set state: `actionPlan: ActionPlan | null`
    4. Render `<ActionConfirmation plan={actionPlan} ... />` inside the command bar dropdown, replacing the search results area
    5. On confirm: Call `/api/ai/execute` with `{ action: inputText, firmId, confirmed: true, actionType: plan.actionType, confirmedPayload: editedPayload }`
    6. On success: Show success message in conversation thread ("Task created successfully" + link to record if URL returned)
    7. On cancel: Clear actionPlan state, return to normal command bar view

    Add `actionPlan` state to the command bar component (this is UI-only state for the confirmation flow, distinct from conversation state in the provider).

    Also wire the same flow in the side panel: if the side panel is open and an nl_action is submitted, show ActionConfirmation inline in the panel. The side panel component should also handle actionPlan state.

    **3. Long-running action feedback:**
    For TRIGGER_DD_ANALYSIS and TRIGGER_IC_MEMO, after confirmed execution:
    - Show message: "DD analysis started for [deal name]. This may take up to 60 seconds. Check the DD tab on the deal page for results."
    - Include a link to the deal page in the message

    **Key patterns:**
    - Confirmation always shown before any mutation — no silent execution
    - Editable fields let the GP correct AI interpretation before committing
    - AMBIGUOUS responses prompt for clarification instead of executing
    - Toast for success/error after execution: `const toast = useToast(); toast.success("Task created")` (no destructuring!)
