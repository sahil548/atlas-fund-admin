   ```typescript
   // Fire-and-forget: save metric snapshot (don't block response)
   const today = new Date();
   today.setHours(0, 0, 0, 0);
   prisma.metricSnapshot.upsert({
     where: {
       investorId_entityId_periodDate: {
         investorId,
         entityId: "", // empty string sentinel for aggregate
         periodDate: today,
       },
     },
     create: {
       investorId,
       entityId: null,
       periodDate: today,
       irr,
       tvpi: metrics.tvpi,

       rvpi: metrics.rvpi,
       nav: currentNAV,
       totalCalled,
       totalDistributed,
     },

       irr,
       tvpi: metrics.tvpi,
       dpi: metrics.dpi,
       rvpi: metrics.rvpi,
       nav: currentNAV,
       totalCalled,
       totalDistributed,
     },
   }).catch((err) => console.error("[metric-snapshot] save failed:", err));
   ```
   IMPORTANT: The @@unique constraint has entityId as nullable. For the aggregate snapshot (entityId=null), use a sentinel approach: store with entityId set to a special value or handle the null case. Since Prisma @@unique with nullable fields treats each null as unique (violating the "one aggregate per day" intent), change the model to use a non-nullable entityId with a sentinel value "__AGGREGATE__" for the aggregate snapshot. Update the model accordingly:
   - Change `entityId String?` to `entityId String` with default "__AGGREGATE__"
   - Remove the `entity Entity?` relation (or make it optional without the FK by removing the @relation and keeping it as a plain string)

   Actually, simpler approach: keep entityId as a plain String (no relation) and store "__AGGREGATE__" for aggregate snapshots or the actual entityId for per-entity snapshots. Remove the Entity relation from MetricSnapshot to avoid FK constraint issues with the sentinel value. Keep only the Investor relation.
